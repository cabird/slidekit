# SlideKit Documentation Generation Plan

## Context Files to Read First

Before executing this plan, read these files in order:

1. `SLIDEKIT_DESIGN.md` — The full design specification. Covers rationale, architecture, and intended API surface.
2. `SLIDEKIT_IMPLEMENTATION_PLAN.md` — The build plan with milestone structure. Shows what's implemented and what's planned.
3. `slidekit/slidekit.js` — The actual library source. This is the authority for what exists.
4. `slidekit/OVERVIEW.md` — A concise (~120 line) conceptual overview of SlideKit for consumers. Already written.

---

## What We're Building

A documentation system for SlideKit consisting of three artifacts:

1. **An API reference document** (`slidekit/API.md`) — comprehensive, structured, generated by an LLM from the source code and executable examples.
2. **Executable example files** (`slidekit/examples/`) — standalone JS files demonstrating common usage patterns. These are real, runnable code that produce rendered slides. They serve as the source of truth for code snippets in the API doc.
3. **A coverage check script** — validates that every exported function from `slidekit.js` appears in the API doc.

Together with the existing `slidekit/OVERVIEW.md`, a consumer gets: conceptual understanding (OVERVIEW.md) → API knowledge with real examples (API.md) → ability to write code.

---

## Design Decisions and Rationale

### Why not enrich JSDoc in slidekit.js directly?

Considered and rejected. JSDoc is per-function, but the real documentation value is cross-function — showing how `text()`, `below()`, and `layout()` compose together. Rich JSDoc with examples would also bloat the 3200-line source file past 5000 lines, making it expensive for AI context consumption. The existing JSDoc stays terse (types, defaults, one-liners) and serves developers reading the source.

### Why not a parallel enrichment file merged by a build script?

Considered and rejected as premature. This would create a triple-sync problem (source JSDoc + enrichment file + generated output) during a period when the API is still evolving. The merge script itself is non-trivial engineering. The approach may make sense later when the API stabilizes, but adds friction now.

### Why executable examples instead of documentation-embedded code blocks?

Code snippets in a markdown file can drift from reality — nothing validates them. Executable example files are real code that runs against the real library. If the API changes and an example breaks, you know immediately. The LLM doc generator pulls snippets from these files, so the API doc always contains tested code.

### Why LLM-generated documentation instead of hand-written?

The API doc can be regenerated whenever the source or examples change. Non-determinism in prose is acceptable — the *structure* is fixed by a template (see below), and the *code examples* are anchored by the executable files. Wording may shift between regenerations; that's fine. The goal is that a model or human can read the output and have enough information to write code and use the system effectively.

### Why separate tests from documentation examples?

Good tests cover edge cases, error handling, and boundary conditions. Good documentation shows the happy path cleanly. A test trying to be both produces noisy documentation and incomplete test coverage. The existing test suite (`slidekit/test/`) handles correctness validation. The example files handle "here's how you use this."

### Why layout coordinate validation over screenshot comparison for testing?

Screenshot comparison (visual regression) is notoriously flaky — it breaks on OS font rendering differences, browser version bumps, anti-aliasing, and subpixel rendering. Layout coordinate validation is deterministic: same input always produces the same resolved coordinates, on every machine. The existing test suite already follows this approach.

Screenshots from executable examples are still valuable — but for human review and documentation illustration, not as automated test assertions.

## Artifact 1: Executable Examples

### Location

`slidekit/examples/` — one JS file per usage pattern.

### Structure of Each Example File

Each example file is a self-contained ES module that:

1. Imports from `slidekit.js`
2. Calls `init()` with appropriate config
3. Builds one slide using SlideKit primitives
4. Calls `layout()` and/or `render()`

The code should be minimal — just the SlideKit calls that demonstrate the pattern. No test assertions, no boilerplate beyond what's needed to run.

Each file should have a brief comment header identifying what pattern it demonstrates.

### Example File List

Target 8–12 examples covering the major usage patterns. Each example should demonstrate a *pattern*, not a single function. Suggested list (adjust based on what the source code actually supports):

| File | Pattern | Key Functions Exercised |
|------|---------|------------------------|
| `title-slide.js` | Full-bleed background image with centered title, subtitle, and accent rule | `image`, `text`, `rule`, `below`, `init`, `layout` |
| `card-row.js` | Horizontal row of styled cards with equal sizing | `hstack`, `rect`, `text`, `group` or `panel`, styling via `style` and `className` |
| `bullet-list.js` | Slide with heading, accent rule, and formatted bullet list | `text`, `rule`, `bullets`, `below`, `vstack` |
| `relative-positioning.js` | Elements positioned relative to each other with gaps | `text`, `below`, `rightOf`, `alignTopWith`, `centerHWith` |
| `text-fitting.js` | Auto-sizing text to fit a bounding box, with measurement | `measureText`, `fitText`, `text` with `overflow` policies |
| `alignment-distribution.js` | Aligning and distributing elements across the slide | `rect`, `alignTop`, `distributeH`, `matchWidth`, `matchSize` |
| `safe-zone-validation.js` | Demonstrating safe zone checks and layout validation | `init` with custom `safeZone`, `layout`, inspecting `warnings` and `errors` |
| `connectors.js` | Flowchart-style diagram with connected boxes | `rect`, `text`, `connect` |
| `grid-layout.js` | Using the grid system for consistent multi-slide positioning | `grid`, `text`, `rect`, grid helpers |
| `full-presentation.js` | A minimal 3-slide presentation rendered end-to-end | `render`, multiple slide definitions, `notes`, `background` |

**Important:** Only create examples for features that actually exist in `slidekit.js`. Check the source before creating each example. If a feature from the design doc isn't implemented yet, skip it.

### Running Examples

The examples need a browser environment (DOM-based text measurement, font loading, rendering). Two approaches:

1. **Manual/AI-driven:** An AI agent (or human) opens a harness HTML page that loads an example, renders it, and optionally screenshots via Playwright. This is the expected workflow when generating documentation.

2. **Harness page:** A simple HTML file (`slidekit/examples/run.html`) that can load any example by query parameter (e.g., `run.html?example=title-slide`), render the slide full-viewport, and display it. This makes it easy to view and screenshot examples.

The harness does not need to be complex — a minimal HTML page that imports the example module and calls its render function.

## Artifact 2: API Reference Document

### Location

`slidekit/API.md`

### Generation Process

An LLM reads:
- `slidekit/slidekit.js` — for all exported function signatures, parameter types, defaults, and return shapes
- `slidekit/examples/*.js` — for real code snippets to embed
- `SLIDEKIT_DESIGN.md` — for rationale and context (but the source code is authoritative for what exists)

The LLM generates `slidekit/API.md` following the fixed outline below.

### Fixed Outline (Template)

The API doc must follow this structure. The LLM can adjust prose between regenerations, but the section structure must remain stable:

```
# SlideKit API Reference

## Prerequisites
(Read OVERVIEW.md first, init/config, font loading)

## Element Primitives
(text, image, rect, rule, group — signatures, all properties with types and defaults, what each returns)

## Text System
(measureText, fitText, overflow policies — how text measurement works, auto-fitting behavior)

## Positioning
### Anchor System
(the 9-point anchor model, resolveAnchor)
### Relative Positioning Helpers
(below, above, rightOf, leftOf, centerVWith, centerHWith, alignTopWith, alignBottomWith, alignLeftWith, alignRightWith, centerIn)
### Stack Layouts
(vstack, hstack — how they resolve to absolute positions)

## Alignment & Distribution
(alignLeft/Right/Top/Bottom/CenterH/CenterV, distributeH/V, matchWidth/Height/Size, fitToRect)

## Styling
(CSS pass-through via style property, blocked properties, className, convenience props vs style)

## Layout & Validation
(layout() — what it takes, what it returns, the scene graph structure, warnings, errors, collisions)
(safeRect, getConfig, validation levels)

## Rendering
(render() — how it produces DOM output, Reveal.js integration)

## Compound Primitives
(bullets, connect, panel — if implemented)

## Additional Features
(grid, snap, percentage sugar, rotate, repeat — if implemented)

## Configuration Reference
(full init() config object, all options)

## Usage Patterns
(embed code from executable examples, with brief narrative for each pattern)
```

### Guidelines for the LLM Generator

- **Only document what exists in the source code.** Do not document planned features from the design doc that aren't implemented.
- **Every exported function must appear.** Cross-reference against the export list.
- **Embed code snippets from the example files**, not invented examples. Reference the example file name so a reader can find and run the full example.
- **Property tables** for each function: property name, type, default value, description. Pull defaults from the source code's `applyDefaults` calls.
- **Keep it practical.** An AI model reading this should be able to write a working slide definition without guessing. A human should be able to scan it and find what they need.
- **Proportional coverage.** Spend more space on `layout()`, `text()`, `measureText()`, and `fitText()` (complex, many options). Less on `matchWidth()` or `rule()` (simple, obvious).
- **Include the return shape** for functions that return structured data (`layout()`, `measureText()`, `fitText()`). Show the actual object structure the caller gets back.

## Artifact 3: Coverage Check Script

### Location

`slidekit/scripts/check-doc-coverage.sh` (or `.js` if preferred)

### What It Does

1. Extracts all `export function` names from `slidekit/slidekit.js`
2. Checks that each function name appears in `slidekit/API.md`
3. Reports any missing functions
4. Exits with non-zero status if coverage is incomplete

### What It Doesn't Do

- It doesn't validate that the documentation is *correct* — just that every exported function is mentioned.
- It doesn't check example files for coverage (that's a judgment call about which patterns matter, not a mechanical check).

## Workflow: Generating Documentation

When it's time to generate or regenerate the API doc:

1. **Verify examples run.** Open `slidekit/examples/run.html` in a browser (or via Playwright) and confirm each example renders without errors.
2. **Optionally capture screenshots.** For each example, screenshot the rendered output. These can be stored alongside the examples or in the doc.
3. **Run the LLM doc generator.** Provide the LLM with: `slidekit.js`, all example files, `SLIDEKIT_DESIGN.md`, and this plan (for the fixed outline and guidelines). The LLM produces `slidekit/API.md`.
4. **Run the coverage check.** Execute the coverage script to verify no functions were missed.
5. **Human review.** Skim the generated doc for obvious errors or gaps. Check that the embedded example snippets match the actual example files.

---

## What's Out of Scope

- **Automated visual regression testing** — Screenshots are for human review and documentation illustration, not automated pixel comparison. Layout coordinate validation in the existing test suite handles correctness.
- **Documentation build pipeline / CI integration** — Documentation generation is AI-driven and on-demand, not a CI step. This may change later.
- **API doc versioning** — The doc reflects current source. No multi-version support.
- **Interactive documentation (Storybook, etc.)** — The example harness is minimal. Rich interactive docs are a future consideration.
